# 🩺 Shadow VIP — نسخة التشخيص الكاملة (Full Runtime Diagnostic Build)

> **الغرض:** تسجيل **خط زمني كامل** لكل ما يستطيع اللودر والبيئة الافتراضية ملاحظته — من **`Application.onCreate`** حتى الخروج — لنعرف **بالدليل الحي** أين تتوقف عملية تسجيل الدخول (Facebook / Twitter / Google) بدل التخمين. **اختبار واحد يجمع كل الأدلة.**

الملف: **`Shadow_VIP_DIAG.apk`** — الوسم في السجل: **`PUBGDIAG`**

صيغة كل سطر: `#<تسلسلي> [<HH:mm:ss.SSS>] S<جلسة> <الحدث> <تفاصيل>`

---

## ✅ ما تسجّله (الخط الزمني الكامل)

| المرحلة | الوسم في السجل | ماذا يعني |
|---|---|---|
| بدء التطبيق | `APP_attachBaseContext` / `APP_onCreate` | أول نقطة تشغيل للودر |
| تحميل المكتبة الأصلية | `LOADLIB_boxapp_ok` | نجاح تحميل libclient (النظام الأصلي) |
| تشغيل تطبيق الضيف | `GUEST_callApplicationOnCreate` | بدء عملية PUBG داخل البيئة الافتراضية |
| دورة حياة النشاط | `ACT_onCreate <class>` | كل Activity تُفتح (بما فيها شاشات OAuth) |
| تثبيت/تشغيل داخل VA | `VA_INSTALL <pkg>` / `VA_LAUNCH <pkg>` | نسخ/تشغيل تطبيق داخل البيئة |
| كل startActivity | `STARTACT act=... dat=... cmp=... pkg=...` | كل نية + الرابط الكامل |
| 🎯 حجب رابط الويب | `BLOCK_HTTP_VIEW_RETURN_-1 dat=https://...` | **الدليل الحاسم**: وصل رابط ويب وحُجب بـ return -1 |
| عودة الـcallback | `onNewIntent act=... dat=...` | 🎯 عودة OAuth عبر deep-link إلى اللعبة |
| حلّ النوايا | `QUERY_ACT` / `RESOLVE_ACT` / `RESOLVE_SVC` | البحث عن متصفح/خدمة لفتح الرابط |
| ربط الخدمة | `BINDSVC act=... pkg=...` | ربط CustomTabs بالمتصفح |
| فحص الحزمة/التوقيع | `GETPKGINFO pkg=...` | استعلام facebook/twitter/gms |
| هوكات الدخول | `FBHOOK_beforeHook_FIRED` / `WEBVIEWHOOK_beforeHook_FIRED` | هل تعمل فعلاً (تحسم "الكود الميت") |
| الأخطاء | `... EX=` + StackTrace كامل | أي استثناء بأثر مكدس كامل |
| ملخص النهاية | `===== SESSION SUMMARY :: events=.. loginSessions=.. startActivity=.. blockedHttpView=.. onNewIntent=.. exceptions=.. uptimeMs=..` | إحصاء ما حدث + زمن التشغيل الكلي |

**ميزات:** توقيت لكل سطر ✅ — رقم جلسة لكل محاولة دخول ✅ — StackTrace كامل ✅ — عدّادات + ملخص نهائي ✅ — حفظ في ملف + مشاركة ✅

## 📤 زر التصدير / المشاركة (تلقائي بعد الخروج من اللعبة)

عند **العودة من PUBG إلى اللودر** (شاشة الإعدادات الرئيسية)، إذا حدثت محاولة تسجيل دخول، سيقوم التطبيق تلقائياً بـ:
1. توليد **ملخص الجلسة**.
2. تجميع كل سطور `PUBGDIAG` (عبر `logcat -d` من نفس هوية العملية — يلتقط سطور الضيف بموثوقية، دون التأثر بإعادة توجيه ملفات البيئة الافتراضية).
3. حفظها في ملف وعرض **Toast بمساره الكامل**:
   ```
   /sdcard/Android/data/com.pubgm/files/pubg_login_export.txt
   ```
4. فتح **نافذة مشاركة** مباشرة (واتساب/تيليجرام/بريد) لإرسال الملف لي بضغطة.

> يظهر هذا **مرة واحدة** بعد أول محاولة دخول في كل تشغيل. أرسل لي الملف الناتج.

## 📲 خطوات الاختبار

1. احذف النسخة الحالية وثبّت `Shadow_VIP_DIAG.apk`، وسجّل الدخول (مفتاح: `ali2026` / `vip`).
2. افتح PUBG، اذهب لتسجيل الدخول، وجرّب **Facebook** (مع تطبيق فيسبوك مثبّت، ثم بدونه)، ثم **Twitter**، ثم **Google**.
3. ارجع إلى اللودر → ستظهر نافذة المشاركة تلقائياً → أرسل لي `pubg_login_export.txt`.

### الطريقة البديلة الأضمن (كمبيوتر + adb):
```bash
adb logcat -c
adb logcat -v time PUBGDIAG:V *:S > pubg_login.txt
```

## 🔴 حدود صادقة
- تسجّل **كل ما يعبر حدود البيئة الافتراضية** (نية/حزمة/خدمة/نشاط) — وهذا مكان الإجابة غالباً.
- **لا تصل** إلى ما بداخل كود Facebook/Twitter SDK نفسه (منطق OAuth الداخلي) لأنه داخل PUBG المحجوب، ولا إلى تفاصيل JNI الأصلية — نراها فقط عبر ما تستدعيه من النظام.

## 🔎 كيف نقرأ النتيجة
- ظهور `BLOCK_HTTP_VIEW_RETURN_-1 dat=https://...facebook/twitter...` = **إثبات قاطع** أن الحجب هو السبب.
- عدم ظهور `STARTACT` بـ VIEW/https أثناء الدخول = PUBG يستخدم مساراً آخر (WebView داخلي/SDK) والحجب ليس السبب.
- ظهور `onNewIntent` بعد الدخول = الـcallback عاد؛ غيابه = الطلب خرج ولم يعُد.
- ظهور `FBHOOK/WEBVIEWHOOK` = الهوك ليس ميتاً؛ غيابه يؤكد أنه ميت.

أرسل الملف وسأحلّل الخط الزمني سطراً سطراً وأحدّد نقطة التوقف بالدليل.
