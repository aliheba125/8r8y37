# تقرير الفحص الديناميكي الكامل — Zero Loader V4.4.01 (بدون روت، عبر تعديل تخطّي المفتاح)

> **الهدف:** تشغيل التطبيق بالكامل داخل محاكي معزول بدون مفتاح ترخيص، وفتح القائمة العائمة وتفعيل كل الخيارات، ومراقبة ماذا يُنزَّل ويُعدَّل وكيف يحمي نفسه، وتحديد سلوكه **بدون روت** والفرق بين الوضعين — كل ذلك للدراسة الدفاعية فقط.
>
> **حالة كل معلومة أدناه: ✅ مُتحقَّق منها حيًّا على المحاكي** ما لم يُذكر خلاف ذلك.

## 0. التعديل الذي أُجري (تخطّي المفتاح فقط)

عُدِّل **دالّة واحدة فقط**: معالِج زر «Login Now» في `LoginActivity` (‏`lambda$InitView$4`). بدل إرسال المفتاح إلى Firebase للتحقّق، أصبح الزر:

```
heis = true
Intent i = new Intent(this, MainActivity)
i.putExtra("Wolf", "BYPASS")     // يتخطّى فحص التوقيع VERSIONS()
i.putExtra("EXP", "20991231235959")  // تاريخ انتهاء بعيد
startActivity(i); finishActivity(0)
```

ثم أُعيد بناء الـ APK بـ apktool وأُعيد توقيعه.

**لم يُلمَس أي شيء آخر:** لا تنزيل الحمولة، ولا قراءة الذاكرة، ولا الخطف، ولا القائمة العائمة، ولا الأوضاع. 
**تأكيد أنه لم يتوقّف/يختفِ شيء:** بعد التعديل عملت كل الوظائف كما هي (التنزيل، تحميل المكتبات الأصلية، القائمة العائمة، العدّادات، الأوضاع) دون أي انهيار — انظر الأقسام التالية.

> ملاحظة: هذا الـ APK المعدَّل **أداة اختبار داخلية فقط ولم يُضَف إلى المستودع**؛ المُضاف هو التطبيق الأصلي قبل التعديل (`Zero_LoaderV4.4.01.apk`).

## 3. السلوك بدون روت — أهم نتيجة

- الدالّتان `loadAssets()/loadAssets2()` تنفّذان `Runtime.exec("chmod 777 " + sock/bypass)`.
- **بعد التنفيذ بقيت صلاحيات الملفات `-rw-------` (600) ولم تصبح 777** ⇒ بدون روت لم تُمنَح صلاحية التنفيذ لِـ `sock`/`bypass`.
- **النتيجة:** لا يمكن تشغيل شيطان `sock` (قارئ `/proc/pid/mem`) ولا تطبيق رقعة `bypass` على `libanogs.so` كعمليات خارجية دون روت. أي أنّ **تجاوز مكافحة الغش على مستوى النظام + قراءة ذاكرة اللعبة عبر الشيطان يتطلّبان روت**.
- ما **يعمل بدون روت:** الواجهة الكاملة، القائمة العائمة، رسم الطبقة العائمة (Overlay)، وحاوية BlackBox الافتراضية — لكن دون رفع صلاحيات فعلي على لعبة حقيقية.

## 5. الوضعان Safe / Brutal — الفرق مُفكَّك من الكود ومُختبَر

`RadioGroup groupmode` يضبط الثابت `MainActivity.kernel`:

| التبويب | الراديو | `kernel` | الأثر في القائمة العائمة (`FloatLogo`) |
|---|---|---|---|
| **Safe** | `system` | `true` (الافتراضي) | `bg_trackBtn` **مخفي** — لا زر تتبّع/أيمبوت |
| **Brutal** | `kernel` | `false` | `bg_trackBtn` **ظاهر** — يكشف زر التتبّع (الأيمبوت) العدواني |

```java
if (MainActivity.kernel) { bg_trackBtn.setVisibility(GONE); }
else                     { bg_trackBtn.setVisibility(VISIBLE); }   // Brutal
```

- الوضع مرتبط أيضًا بزر `installKernelButton` وقائمة `kerneldriver` ⇒ خيار وصول للذاكرة عبر **مُشغّل نواة (kernel driver)** يتطلّب روت/وحدة نواة.
- **مُختبَر:** الضغط على Brutal يُبرزه ويظلّ التطبيق مستقرًّا. **الخلاصة:** Brutal ليس «أقوى غشًّا» بحد ذاته بل **يكشف تبويب/زر الأيمبوت (Track) ويفعّل مسار قراءة ذاكرة أعمق (نواة)**؛ Safe يخفيه ويستخدم مسارًا أخفّ.

## 6. الحماية الذاتية (كما لُوحظت حيًّا)

- **`VERSIONS()`** فحص توقيع/سلامة أصلي — يُتخطّى فقط عند وجود `Wolf=BYPASS` في الـ Intent (وهو مسار الدخول الشرعي نفسه)، لذلك إعادة توقيع الـ APK لم تُطلِق الشاشة الصفراء.
- **`BYPASS()`** يُستدعى عند `heis=true` → رقعة أصلية على نظام مكافحة الغش (`libanogs.so`) — تتطلّب روت لتطبيقها فعليًا.
- **BlackBox** (`com.pubgm:black`) عملية ثانوية دائمة (حاوية افتراضية لعزل/إخفاء الروت) — بقيت حيّة ومستقرّة طوال الجلسة.
- مفاتيح **swAntiCrash** و **swHideRoot** في الإعدادات (منع الانهيار + إخفاء الروت).

---

## 7. «هل يكتشف إن كنتَ اللعبة؟»

لا. المُحمِّل يعامل اللعبة كحزمة منفصلة (`com.tencent.ig`…) يستضيفها داخل BlackBox، ويجدها عبر `pidof` ثم يقرأ `/proc/pid/mem`. على المحاكي بلا لعبة مثبَّتة: العدّادات تقرأ 0 ولا هدف ذاكرة، و**لا ينهار** — ببساطة لا يجد ما يقرؤه. فهو يكتشف **عملية اللعبة بالاسم** ويعمل عليها، ولا «يكتشف أن الجهاز هو اللعبة».

## 8. لا شيء تدميري ولا انهيار

- طوال الجلسة الكاملة في `logcat`: **لا `rm -rf`، لا `reboot`، لا `wipe`، ولا `FATAL/ANR`** لحزمة `com.pubgm`.
- البيانات سليمة، التطبيق بقي على `MainActivity`، والعمليتان (`com.pubgm` و`com.pubgm:black`) حيّتان.
- يؤكّد هذا استنتاج التحليل الساكن السابق: **النسخة الحالية لا تحوي مفتاح تدمير**.

# 10. سيناريو اللعبة الوهمية (Mock Game) — مراقبة strace منخفضة المستوى

> **الجديد في هذه المرحلة:** بدل تشغيل التطبيق بلا لعبة، أنشأنا **لعبة PUBG وهمية** (‏`com.tencent.ig` v4.4، حزمة stub بمكتبات 64-بت + ملف OBB وهمي) بحيث يظنّ Zero Loader أنّ اللعبة موجودة، ثم شغّلنا المسار الكامل (Install Game → OK → Play → Start Menu → ESP/Items/WideView/Skins → Safe/Brutal) تحت مراقبة **`strace -f -tt -T`** على العمليتين `com.pubgm` و`com.pubgm:black` وعملية اللعبة داخل الحاوية، مع مقارنة لقطات قبل/بعد.
>
> **حالة كل بند: ✅ مُتحقَّق منه حيًّا بأدلّة strace/الملفات** ما لم يُذكر خلاف ذلك.

## 10.0 البيئة والتوافق (ABI)

- المحاكي `zerotest` (API 30، x86_64) لكنّ `com.pubgm` مثبّت بـ `primaryCpuAbi=arm64-v8a` ويُشغَّل **مُترجَمًا كـ arm64** عبر `/system/lib64/libndk_translation.so` (‏`ro.product.cpu.abilist = x86_64,x86,arm64-v8a,…`).
- عملية اللعبة داخل الحاوية `com.tencent.ig` تعمل عبر `app_process64` وبنفس مسار الترجمة، لذلك **مكتبات الغش arm64 متوافقة وتُحمَّل بنجاح** (‏`libclient.so` و`libTcore.so` من `lib/arm64/` ظهرتا في `/proc/<pid>/maps` لكل من المُحمِّل وعملية اللعبة). **لا يوجد تعارض ABI.**

## 10.1 كشف اللعبة والتثبيت داخل الحاوية (لا PackageManager النظام)

- الكشف يتمّ داخل بيئة TCore/BlackBox الافتراضية لا عبر نظام أندرويد. عند الضغط على **INSTALL GAME** لـ PUBG GLOBAL ظهر BottomSheet «Client: PUBG GLOBAL — This process may take 1-3 minutes…».
- عند **OK** أُنشئت حاوية افتراضية كاملة داخل التخزين الخاص بالمُحمِّل (وليس في النظام):

| المسار الجديد (بعد التثبيت في الحاوية) | المحتوى |
|---|---|
| `files/data/app/com.tencent.ig/package.conf` (5308 بايت) | سجلّ TCore للحزمة: `ApplicationInfo`, شهادة التوقيع، مسارات lib (‏`x86_64` و`arm64-v8a`)، `targetSdk=30`, `versionName 4.4` (فئات `com.tcore.core.system.pm.BPackage`) |
| `files/data/app/com.tencent.ig/lib/` | دليل مكتبات الحاوية |
| `files/data/user/0/com.tencent.ig/{files,shared_prefs,no_backup}` | دلائل بيانات اللعبة الافتراضية |
| `files/data/user_de/0/com.tencent.ig/` | بيانات الجهاز المشفّرة الافتراضية |
| `files/system/{uid.conf,user.conf,shared-user.conf}` | بيانات إدارة حِزَم TCore |

- **ملاحظة مهمة:** TCore **لا ينسخ APK اللعبة**؛ `package.conf` يشير مباشرةً إلى `base.apk` الـ stub المثبّت على النظام ويحاكي بقيّة بيئة اللعبة (دلائل بيانات + توقيع + ABI). بعد OK تحوّل الزرّ إلى **PLAY GAME** ⇒ `isInstalled(pkg)` صار `true`.

## 10.2 تشغيل اللعبة (Play) — عملية بنفس الـ uid

- الضغط على **PLAY GAME** ⇒ `TCoreCore.launchApk()` أنشأ عملية جديدة **`com.tencent.ig`** بنفس مستخدم المُحمِّل `u0_a168` (‏`ps`: `u0_a168 … com.tencent.ig`).
- `/proc/<gamepid>/exe → /system/bin/app_process64`، و`/proc/<gamepid>/maps` تُظهر تحميل `libTcore.so` و`libclient.so` (مكتبات المُحمِّل) داخل عملية اللعبة الوهمية ⇒ **الحقن داخل العملية يتمّ عبر الحاوية بنفس الـ uid دون روت**.
- عملية اللعبة نفسها (stub) خاملة: `game.strace` لا يحوي `ptrace`/`process_vm`/socket (اللعبة الوهمية بلا منطق).

## 10.3 تصحيح جوهري لِما ورد في القسم 3: **`chmod 777` و`sock` ينجحان بلا روت**

هذا أهمّ اكتشاف جديد ويُصحّح استنتاجًا سابقًا. القسم 3 ذكر أنّ `chmod 777` يفشل وتبقى الصلاحيات 600؛ ذلك كان مبنيًّا على حالة **قبل تشغيل لعبة**. عند تشغيل مسار Play/Start Menu الكامل ظهر التالي في `strace`:

```
9687  09:39:42.757069 execve("/data/user/0/com.pubgm/files/sock", ["…/sock"…], …) = 0   ← الشيطان اشتُغِّل فعلاً
…
sock الآن:  -rwxrwxrwx 1 u0_a168 u0_a168 43784 …   ← صار 777 (كان 600)
```

- التطبيق يشغّل `Runtime.exec("chmod 777 …/sock")` عبر `/system/bin/chmod`، و**التطبيق يملك الملف فينجح `chmod` دائمًا** (لا يحتاج روت لتعديل صلاحيات ملفاته). ثمّ **`execve("…/sock")` نجح** = 0 ⇒ **الشيطان `sock` يُنفَّذ بلا روت.**
- إذًا العائق الحقيقي **ليس** `chmod` ولا تشغيل الشيطان، بل **مرحلة قراءة الذاكرة** (القسم التالي).

> `bypass` (رقعة `libanogs`) بقي `600` لأنّ مساره لم يُطلَب في هذا السيناريو (لا مكافحة غش حقيقية في اللعبة الوهمية).

## 10.4 العائق الحقيقي بلا روت: `ptrace` يفشل بـ EPERM

بعد تشغيل الشيطان `sock`، حاول قراءة/خطف الذاكرة عبر `ptrace`، والنتيجة قاطعة:

```
9701  09:39:44.929713 ptrace(PTRACE_SEIZE, 9324, NULL, 0) = -1 EPERM (Operation not permitted)
9701  …            ptrace(PTRACE_INTERRUPT, 93xx) = -1 ESRCH
9701  …            ptrace(PTRACE_DETACH,    93xx) = -1 ESRCH
```

| النداء | عدد المرّات | النتيجة |
|---|---|---|
| `ptrace(PTRACE_SEIZE, <tid>)` | **29** | **EPERM** (ممنوع) |
| `ptrace(PTRACE_INTERRUPT)` | 28 | ESRCH |
| `ptrace(PTRACE_DETACH)` | 28 | ESRCH |
| `prctl(PR_SET_PTRACER, PR_SET_PTRACER_ANY)` | — | EINVAL |

- الشيطان عدّد **كل خيوط المُحمِّل** (‏tid من 9324 إلى 9474) وحاول `PTRACE_SEIZE` على كلٍّ منها ⇒ **كلّها EPERM**. سبب المنع: سياسة SELinux لنطاق `untrusted_app` + `Yama ptrace_scope` تمنع الخطف بين العمليات حتى بنفس الـ uid دون روت.
- مباشرةً بعد الفشل اتّصل الشيطان بـ `/dev/socket/tombstoned_crash` (‏سجل انهيار) ⇒ **الشيطان انهار بعد فشل الخطف** (مرّة واحدة `tombstoned_crash`).
- `process_vm_readv` الوحيدة التي نجحت (15 مرّة) كانت **قراءة ذاتيّة** (‏`process_vm_readv(6345, …)` حيث الهدف = العملية نفسها) — مسموحة دائمًا ولا تعني قراءة ذاكرة عملية أخرى.

**الخلاصة المصحّحة:** بلا روت ⇒ `chmod`+تشغيل `sock` **ينجحان**، لكن **قراءة/خطف ذاكرة أي عملية عبر `ptrace` تفشل (EPERM)** ⇒ الشيطان الخارجي عاجز عن قراءة ذاكرة اللعبة. الغش الذي يعمل بلا روت هو ما يمرّ **داخل عملية اللعبة نفسها** (‏`libclient` المحقونة في الحاوية) لا عبر الشيطان الخارجي.

## 10.9 ما نجح مقابل ما فشل بلا روت (خلاصة المرحلة)

| العملية | بلا روت | الدليل |
|---|---|---|
| الدخول (تخطّي المفتاح) + الواجهة الكاملة | ✅ يعمل | `MainActivity` بلا شاشة صفراء |
| تنزيل الحمولة (`servernrot.zip`→`sock`/`bypass`/`libpubgm`) | ✅ يعمل | الملفات + sha256 مطابق |
| تثبيت اللعبة داخل حاوية TCore | ✅ يعمل | `package.conf` + دلائل الحاوية |
| تشغيل اللعبة بنفس الـ uid + حقن `libclient`/`libTcore` داخل العملية | ✅ يعمل | `maps` عملية اللعبة |
| `chmod 777` + `execve(sock)` (تشغيل الشيطان) | ✅ **ينجح** (تصحيح) | strace `execve … = 0`، sock=777 |
| القائمة العائمة + Overlay + HUD + حفظ الإعدادات | ✅ يعمل | `FloatLogo`/`Overlay` + `espValue.xml` |
| **قراءة/خطف ذاكرة عملية أخرى عبر `ptrace`/الشيطان** | ❌ **يفشل** | `PTRACE_SEIZE = EPERM` ×29 ثمّ انهيار |
| رقعة `libanogs` عبر `bypass` كعملية خارجية | ❌ يتطلّب روت | `bypass` بقي 600 (لم يُطلَب) |

**الاستنتاج النهائي للمرحلة:** المحاكاة الافتراضية (TCore/BlackBox) تتيح **تثبيت اللعبة وتشغيلها وحقن مكتبات الغش داخل عمليتها بنفس الـ uid بلا روت**، لكنّها **لا تتيح قراءة ذاكرة عملية أخرى عبر الشيطان الخارجي** لأنّ `ptrace` ممنوع بـ EPERM دون روت. الغش الذي يعمل بلا روت محصور بما يُنفَّذ **داخل عملية اللعبة نفسها** عبر المكتبة المحقونة. لا أوامر تدميرية ولا انهيار للتطبيق طوال السيناريو.
